## 二、🔴 Critical 问题

### C1. 商品成色（condition）枚举值前后端与数据库不匹配

**问题描述：** 前端发送英文枚举值，数据库 CHECK 约束只接受中文值，后端未做转换。

| 层级 | 字段/参数 | 可选值 |
|---|---|---|
| **Frontend** `Goods.condition` | `'new' \| 'like_new' \| 'slight_use' \| 'obvious_trace'` |
| **Backend** `CreateGoodsRequest.Condition` | `string`（无校验） |
| **Database** `goods.goods_condition` | `CHECK IN ('全新', '几乎全新', '轻微使用', '明显痕迹')` |

**影响：** 前端调用 `createGoods` 或 `updateGoods` 时，后端 [GoodsRepository.CreateAsync](backend/CampusTrade/CampusTrade.Backend/Repositories/GoodsRepository.cs#L123-L128) 直接将英文值 `"new"` 写入数据库，触发 Oracle CHECK 约束错误（ORA-02290），**商品发布/编辑功能完全不可用**。

**修复建议：** 在 Backend Service 层或 Repository 层添加枚举映射：

```
"new"           → "全新"
"like_new"      → "几乎全新"
"slight_use"    → "轻微使用"
"obvious_trace" → "明显痕迹"
```

契约文档 [backend-rebuild-contract.md](docs/backend-rebuild-contract.md#L707-L715) 第 9 节已明确标注此映射，但后端未实现。

### C2. 登录接口响应体字段冗余

**问题描述：** 后端 `LoginHttpResponseDto` 同时将 `Token` 和 `User` 放在响应根级别和 `Data` 内部，导致响应体结构混乱。

```csharp
// backend/.../Models/DTOs/AuthDto.cs:60-78
public class LoginHttpResponseDto
{
    public int Code { get; set; } = 200;
    public string Message { get; set; } = "success";
    public AuthResponseDto Data { get; set; } = new();   // Data.Token, Data.User
    public string Token { get; set; } = string.Empty;     // 根级别重复！
    public UserDto User { get; set; } = new();            // 根级别重复！
}
```

**实际返回的 JSON：**
```json
{
  "code": 200,
  "message": "success",
  "data": { "token": "xxx", "user": {...} },
  "token": "xxx",        // ← 冗余
  "user": {...}          // ← 冗余
}
```

**影响：** 前端 `request.post<LoginResult>('/auth/login', params)` 通过 axios 响应拦截器拿到 `response.data`（即上方的整个 JSON），但前端 `LoginResult` 类型定义为 `{ token, user }`。如果页面代码直接读 `res.data.token`，实际拿到的是根级别的 `token`（意外能工作），但这属于巧合，数据结构不规范。

**修复建议：** 删除 `LoginHttpResponseDto` 中根级别的 `Token` 和 `User` 属性，统一使用 `ApiResponse<AuthResponseDto>` 格式。

### C3. 前端 request 拦截器不自动解包 `data`

**问题描述：** 前端 axios 响应拦截器在 `code === 200` 时直接 `return response`（完整的 AxiosResponse），而非 `return response.data`。

```typescript
// frontend/src/utils/request.ts:52-68
service.interceptors.response.use(
  (response: AxiosResponse<ApiResponse>) => {
    const res = response.data
    if (res.code !== 200) { ... }
    return response   // ← 返回 AxiosResponse，不是 ApiResponse<T>.data
  }
)
```

**影响：** 调用方 `request.get<PageResult<Goods>>('/goods')` 实际拿到的是：
```json
{
  "code": 200,
  "message": "success",
  "data": { "list": [...], "total": 10, "page": 1, "size": 10 }
}
```
而非期望的 `{ list: [...], total: 10, page: 1, size: 10 }`。

如果前端页面代码写 `res.data.list`，实际访问的是 `ApiResponse.data.list`，尚能工作。但如果写 `res.list` 则会拿到 `undefined`。这取决于各页面的实际取值方式，存在不一致风险。

**修复建议：** 两种方案选其一：

- **方案A（推荐）：** 拦截器返回 `res.data`（即完整 `ApiResponse<T>`），页面统一用 `res.data.list`
- **方案B：** 拦截器返回 `res.data`（直接解包到 `T`），页面直接用 `res.list`

建议统一为方案 B，与契约文档 `backend-rebuild-contract.md` 第 3 节描述的「统一解包为业务 data」一致。

### C4. 前端 User 类型包含 `password` 字段

**问题描述：**

```typescript
// frontend/src/types/index.ts:1-13
export interface User {
  userId: number
  username: string
  password: string      // ← 安全隐患
  ...
}
```

**影响：** 后端 `AuthService.ToDto()` 已正确将 `Password` 设为 `string.Empty`，但前端类型定义暗示密码可能出现在响应中，容易导致开发者误用。前端页面如果渲染用户信息时遍历 User 字段，可能意外显示密码（即使是空字符串也不应出现该字段）。

**修复建议：** 从 `User` 接口中移除 `password` 字段。注册和登录使用独立的 `LoginParams` / `RegisterParams` 类型（已存在）。

### H2. 后端 Controller 未接入 JWT 认证

**问题描述：** `GoodsController` 中所有需要当前用户身份的接口均使用硬编码值：

```csharp
// GoodsController.cs 多处出现
int currentUserId = 1;    // TODO: 从 JWT 获取
bool isAdmin = false;     // TODO: 检查角色
```

**影响：** 所有商品操作（创建/编辑/删除/审核/下架）都以 `userId=1` 的身份执行，权限校验完全失效。任何人可以修改/删除任何人的商品。

**修复建议：** 参照 `AuthController.ResolveCurrentUserId()` 的实现模式，从 `HttpContext.User` 中解析 JWT Token 获取 `userId` 和 `role`。

## 四、🟡 Medium 问题

### M1. 后端未使用数据库存储过程，自行拼接 SQL

**问题描述：** 数据库已编写 13 个完善的存储过程（含事务控制、异常处理），但后端直接编写 SQL。

| 数据库存储过程 | 用途 | 后端现状 |
|---|---|---|
| `sp_place_order` | 下单（校验→锁商品→关议价→插订单） | ❌ 未调用，后端无订单功能 |
| `sp_audit_goods` | 商品审核（权限校验→更新状态→记日志） | ❌ 未调用，后端手动 UPDATE + INSERT |
| `sp_cancel_order` | 取消订单（校验→恢复商品→取消预约） | ❌ 未调用 |
| `sp_complete_meet` | 完成面交（确认码校验→完成） | ❌ 未调用 |
| `sp_create_bargain` | 创建议价（含自买/重复检测） | ❌ 未调用 |
| `sp_respond_bargain` | 卖家回复议价 | ❌ 未调用 |
| `sp_create_review` | 创建评价 + 更新信用分 | ❌ 未调用 |
| `sp_manage_user_ban` | 封禁用户 + 下架所有商品 | ❌ 未调用 |
| `sp_send_message` | 发送消息 | ❌ 未调用 |
| `sp_get_or_create_session` | 获取/创建聊天会话 | ❌ 未调用 |

**影响：**
- `GoodsRepository.AuditAsync()` 只更新 `goods_status`，不调用 `sp_audit_goods`，导致**审核日志（audit_log）不会被写入**。
- 手动 SQL 缺少存储过程中已实现的权限校验、业务规则检查。
- 未来实现订单等复杂流程时，事务一致性无法保证。

**修复建议：** 交易类、审核类操作统一调用对应存储过程。简单 CRUD 可继续使用直接 SQL。

### M2. `UserHttpResponseDto` 响应格式不标准

**问题描述：** `UserHttpResponseDto` 通过继承 `UserDto` 将响应包装字段与业务字段混在同一层级：

```csharp
public class UserHttpResponseDto : UserDto
{
    public int Code { get; set; }       // 包装字段
    public string Message { get; set; } // 包装字段
    public UserDto Data { get; set; }   // 嵌套自身
    // + 继承自 UserDto 的 UserId, Username, Password, ...
}
```

**返回 JSON 示例：**
```json
{
  "userId": 1,
  "username": "admin",
  "nickname": "管理员",
  "code": 200,
  "message": "success",
  "data": { "userId": 1, "username": "admin", ... }
}
```

**影响：** 与其他接口的 `ApiResponse<T>` 格式不一致，前端解析时需要特殊处理。

**修复建议：** 统一使用 `ApiResponse<UserDto>`，删除 `UserHttpResponseDto` 和 `StudentAuthHttpResponseDto`。

### L2. AuditLog 中 `action` 字段枚举值不统一

**问题描述：**

| 层级 | 值 |
|---|---|
| 前端 `AuditLog.action` | `'approve' \| 'reject' \| 'ban' \| 'offline'` |
| 数据库 SP `sp_audit_goods` 写入 | `p_action` = `'approved'` 或 `'rejected'` |
| 数据库 SP `sp_manage_user_ban` 写入 | `p_action` = `'ban'` 或 `'unban'` |

前端用 `'approve'`（动词原形），存储过程写入 `'approved'`（过去分词），存在不匹配。

**修复建议：** 统一枚举值，建议与存储过程实际写入的值保持一致（`approved`, `rejected`, `ban`, `unban`, `offline`）。

### L3. 数据库视图 `v_goods_detail` 不含 `cover_image` 字段

**问题描述：** `v_goods_detail` 视图没有返回首图字段，但后端 `GoodsRepository.GetByIdAsync()` 中使用 `cover_image AS ImageUrl` 从 `v_goods_detail` 查询，会报错 `ORA-00904: "COVER_IMAGE": invalid identifier`。

查看 [003_views.sql](database/ddl/003_views.sql#L42-L70)，`v_goods_detail` 确实没有 `cover_image` 列。该列仅存在于 `v_goods_list` 中。

**修复建议：** 在 `v_goods_detail` 中添加 `cover_image` 子查询列，或后端 `GetByIdAsync` 改为从 `v_goods_list` 查询（`v_goods_list` 的状态过滤是 `WHERE goods_status IN ('approved', 'sold')`，可能不满足详情页需求），或后端单独 JOIN `goods_image` 表获取首图。
